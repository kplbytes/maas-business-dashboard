# MaaS 商务看板系统 · 技术方案

> 版本：v0.5（全流程审查修订）　日期：2026-07-17
> 定位：**轻量级看板 + 权限控制**。只做两件事——按账号隔离租户数据、展示经营看板。

## 1. 背景与目标

MaaS 平台现有 Metabase 看板（dashboard/14）无法按登录账号做租户级行权限隔离。本系统自建一套轻量看板，让商务同学登录后只看到**自己名下租户**的数据，默认看名下全部租户汇总，与 Metabase 口径对齐。

**目标：**

1. 商务登录默认看「我名下所有租户的汇总」，支持租户/时间筛选。
2. 管理员可对每个商务的租户列表做增删改查。
3. 数据隔离强一致：商务只能看到被分配的租户，越权不可发生。
4. 支持多种图表，整体风格对标 Metabase。
5. **极简**：DB 只存「登录 + 商务-租户关系 + 操作日志」，看板/查询/配置全在代码里，变更靠改代码+部署。

## 2. 解决思路

权限层从 Metabase 搬到**自建后端**——后端是唯一信任边界。所有分析查询执行 SQL 前，服务端强制把「当前用户名下租户集合」注入为 `WHERE tenant_id IN (:tenantIds)`。前端传入的租户筛选只与该集合取交集，绝不作为唯一过滤源。Metabase 仅作设计参考，运行时不依赖。

**单一数据源：** 应用状态（3 表）与分析数据共用同一个 ADB-MySQL 实例，通过**两个数据库账号**隔离权限：
- `app_rw` 账号：对 `users` / `tenant_assignments` / `operation_logs` 3 表有读写权限（管理服务使用）；
- `analytics_ro` 账号：对分析表只读，所有命名 SQL 走此连接（分析查询服务使用）。

看板/查询配置不在 DB，在代码仓库。

### 3.1 前后端分离架构

本系统采用**前后端分离**架构，前端与后端各自独立工程、独立构建、独立部署，通过 RESTful API（JWT 鉴权）通信。

```
┌──────────────────────────────────────────────────────────┐
│  前端工程（独立仓库 / 独立部署）                           │
│  React 18 + Vite + TS + shadcn/ui + Tailwind + ECharts  │
│  构建：npm run build → 静态产物（dist/）                  │
│  部署：Nginx 托管静态资源，反向代理 /api → 后端           │
└────────────────────────┬─────────────────────────────────┘
                         │ HTTPS · RESTful API · JWT
┌────────────────────────▼─────────────────────────────────┐
│  后端工程（独立仓库 / 独立部署）                           │
│  Java 17 + Spring Boot 3.x                               │
│  构建：Maven/Gradle → 可执行 JAR                          │
│  部署：java -jar 或容器化（Docker），独立扩缩容            │
└──────────────────────────────────────────────────────────┘
```

**分离架构带来的发版优势：**

| 场景 | 前端发版 | 后端发版 |
|------|---------|---------|
| UI 调整 / 图表优化 / 新增看板页面 | ✅ 前端独立构建部署，后端不动 | 不涉及 |
| 新增 API / 修改查询逻辑 / 权限规则调整 | 不涉及 | ✅ 后端独立构建部署，前端不动 |
| 看板查询变更（新增 .sql） | 不涉及 | ✅ 后端重新部署加载 |
| 全栈功能上线 | ✅ 前端先发或同时发 | ✅ 后端先发或同时发 |

**关键约定：**
- **接口约定**：前后端共同维护一份接口文档（约 15 个接口，README 级别即可），后端先行定义接口路径和返回结构，前端据此开发，互不阻塞。接口数量小，不引入 Swagger 工具链。
- **版本管理**：前端和后端各自独立 Git 仓库（或 monorepo 下分目录），各自 CI/CD 流水线，独立打 tag 发版。
- **向后兼容**：后端 API 变更遵循向后兼容原则（新增字段不删除字段），前端发版不依赖后端同步发版；如遇 Breaking Change，前后端协调上线顺序。
- **同源部署（重要）**：Nginx 托管前端静态资源并反向代理 `/api/*` → 后端，确保前后端**同源**。JWT 存 httpOnly cookie，同源场景用 `SameSite=Lax`（避免跨域 cookie 丢失导致登录态断裂）；**禁止跨域部署**，否则 cookie 配置复杂且易出 BUG。
- **环境隔离**：前端静态资源由 Nginx 托管，后端只提供 API 服务，二者可部署在不同机器/容器，各自独立扩缩容。

## 5. 数据模型（极简：3 张表）

### 5.1 应用状态库（ADB-MySQL，与分析库同实例）

共 **3 张表**：users（账号）、tenant_assignments（租户分配）、operation_logs（操作日志）。

```
users
  id            BIGINT AUTO_INCREMENT PRIMARY KEY
  username      VARCHAR(64)  UNIQUE      -- 登录名（姓名全拼，重名自增后缀）
  password_hash VARCHAR(255)             -- bcrypt
  display_name  VARCHAR(128)             -- 真实姓名
  region        VARCHAR(64)              -- 所属区域（如 华东区 / 区域A）
  role          VARCHAR(16)              -- ADMIN / SALES（枚举列，仅两角色）
  enabled       BOOLEAN DEFAULT TRUE     -- 启用/停用
  must_reset    BOOLEAN DEFAULT TRUE     -- 首次登录/重置后强制改密
  failed_count  INT DEFAULT 0            -- 连续登录失败次数
  locked_until  TIMESTAMP NULL           -- 锁定截止时间（NULL=未锁定）
  last_login_at TIMESTAMP NULL           -- 最后登录时间
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

tenant_assignments
  id            BIGINT AUTO_INCREMENT PRIMARY KEY
  user_id       BIGINT  →users.id        -- 应用层校验（ADB 不强制外键约束）
  tenant_key    VARCHAR(64)              -- 注入 ADB 的 tenant_id 值
  tenant_name   VARCHAR(128)             -- 冗余存展示名（分配时从 ADB 带回）
  assigned_by   BIGINT  →users.id        -- 应用层校验
  assigned_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  UNIQUE(tenant_key)                     -- 全局唯一：一个租户同时只能分配给一个商务（互斥）
  UNIQUE(user_id, tenant_key)            -- 同一商务不重复分配同一租户

operation_logs
  id            BIGINT AUTO_INCREMENT PRIMARY KEY
  user_id       BIGINT                   -- 操作者ID（NULL=匿名/登录前）
  username      VARCHAR(64)              -- 操作者账号（冗余，便于查询）
  action        VARCHAR(32)              -- LOGIN / LOGOUT / CHANGE_PASSWORD / ASSIGN_TENANT / UNASSIGN_TENANT / CREATE_USER / DISABLE_USER / QUERY
  target        VARCHAR(128)             -- 操作对象（如分配的 tenant_key、创建的用户名）
  detail        TEXT NULL                -- 补充信息（JSON 或文本）
  ip            VARCHAR(45)              -- 请求IP
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  INDEX(user_id), INDEX(action), INDEX(created_at)
```

**username 生成规则（重要）：**
- 登录账号 = 真实姓名的**全拼**（如 张三 → `zhangsan`）。
- 后端使用拼音库（如 `com.github.stuxuhai:jpinyin` 或 `Tinymaxim:pinyin4j`）自动生成。
- **重名自增**：若 `zhangsan` 已存在，则尝试 `zhangsan1`、`zhangsan2`……直到不重复。
- **多音字**：取最常见读音（如「乐」→ `le`），admin 可在创建后手动修改。
- 管理员创建商务时只需输入 `display_name`（真实姓名）+ `region`，`username` 由后端自动生成并返回。

> 角色用 `users.role` 枚举列，不再单独建 roles/user_roles 表。租户主数据**不建表**——分配时 admin 从 ADB 实时查 `SELECT DISTINCT tenant_id, tenant_name` 选取，写入 `tenant_assignments` 时把 key+name 一起存。`tenant_assignments.tenant_key` 的**全局 UNIQUE 约束**确保互斥分配在数据库层强制执行——同一租户不可能同时分配给两个商务。

### 5.2 分析层（ADB-MySQL，只读）

直连 ADB，执行 `resources/sql/*.sql` 命名查询。建议在 ADB 内预聚合宽表（按 `tenant_id/model/date/hour`）保证秒级响应；今日数据走实时表 + 「数据可能滞后」水印。

### 5.3 看板/查询配置（代码，非 DB）

**每个卡片一个独立 SQL 文件**，一一对应，便于单独维护和优化：

```

    total_revenue.sql             -- card 147 消费总金额 · KPI · 含 :tenantIds :startDate :endDate
    total_tokens.sql              -- card 149 Tokens消耗量 · KPI
    total_calls.sql               -- card 148 调用次数 · KPI
    fund_real_balance.sql         -- 真实余额 · KPI · 取 card 161 数据源最新日真实余额
    fund_available_balance.sql    -- 可用余额 · KPI · 取 card 161 数据源最新日可用余额（含授信）
    fund_recharge.sql             -- 充值金额 · KPI · 取 card 161 数据源最新日充值+授信+代金券总额
    tenant_discount.sql           -- card 138 租户折扣率一览 · 表格
    model_usage_detail.sql        -- card 121 模型使用明细 · 表格
    model_token_dist.sql          -- card 9 模型Token消耗分布 · 饼图
    tenant_fund_trend.sql         -- card 161 租户资金行为看板 · 折线图
```

> **布局策略（重要）：** 看板布局由**前端硬编码**（固定 React 组件 + Tailwind 网格），每个组件直接调 `POST /api/analytics/query/{queryName}` 获取数据。不引入 dashboards.yml 配置层——卡片固定，前端组件已知道自己对应哪个 query，多一层配置只会增加维护成本却不带来价值。新增图表 = 前端加组件 + 后端加 `.sql`，两处改动部署即生效。

**时区处理（重要）：**
- 用户侧所有时间筛选默认按**北京时间（Asia/Shanghai, UTC+8）**理解。
- 前端传入的 `:startDate` / `:endDate` 为北京时间日期（如 `2026-07-01`）。
- **SQL 内部做时区转换**：若 ADB 存储的是 UTC 时间，SQL 中用 `CONVERT_TZ(column, '+00:00', '+08:00')` 转换后再过滤。
- 快捷预设（近7天/30天/90天/本月）由前端按北京时间计算起止日期后传入。
- `snapshot_date`（card 161）为日期类型，直接按日期比较，无需转换。

## 6. 权限与安全模型

**认证（无状态 JWT，无 token 表）：**
- 登录签发 access (30min) + refresh (7d)，均为无状态 JWT，存 **httpOnly SameSite=Lax cookie**（同源部署，见 §3.1）。
- 刷新：refresh JWT 验签通过即签新 access；不落库。
- 登出：清 cookie。
- **撤销**：无状态 JWT 不能即时吊销；靠①access 短 TTL（30min 内自然失效）②`users.enabled=false` 阻止新 token 签发。内部工具可接受；若未来需即时踢人再加黑名单表。
- **登录失败限制（落库）**：同一账号连续失败 5 次锁定 15 分钟。计数存 `users.failed_count` + `users.locked_until` 字段（不引入 Redis，只有数据库）：
  - 每次登录失败：`failed_count += 1`，达到 5 次时设置 `locked_until = NOW() + 15min`。
  - 每次登录成功：重置 `failed_count = 0`，`locked_until = NULL`。
  - 登录时先检查 `locked_until > NOW()`，是则拒绝并提示剩余锁定时间。
  - 登录成功后更新 `last_login_at = NOW()`。
  - 写入 `operation_logs`（action=LOGIN）。

**密码加密（bcrypt，非 MD5）：**
- 密码使用 **bcrypt** 哈希存储，不使用 MD5。原因：MD5 可被彩虹表秒破，已不安全；bcrypt 自带盐 + 慢速哈希，是 Spring Security 默认方案，使用也只需一行代码（`BCryptPasswordEncoder.encode()`）。
- 密码修改时，后端校验旧密码 → 通过后对新密码 bcrypt 加密 → 更新 `users.password_hash`。
- **密码修改后强制重新登录**：清 cookie + 前端跳转登录页。无状态 JWT 下旧 token 最多存活 30min 自然失效，但清 cookie 确保当前浏览器立即退出。
- **管理员与商务共用密码修改入口**（侧边栏弹窗），都走此流程。
- 密码修改成功后写入 `operation_logs`（action=CHANGE_PASSWORD）。

**管理员账号初始化：**
- `init_schema.sql` 建表时插入一条默认 admin 记录：`username=admin, password=admin123`（bcrypt hash），`role=ADMIN`，`must_reset=TRUE`。
- 登录时后端返回 `must_reset` 标记，若为 `true` 前端弹窗强制改密，改完才进系统；改密成功后后端置 `must_reset=FALSE`。
- 管理员创建商务账号时设置初始密码（`must_reset=TRUE`），商务首次登录同样强制改密。

**RBAC：** `role=ADMIN` / `role=SALES`，方法级 `@PreAuthorize`。管理端要求 ADMIN，分析端要求已认证。**ADMIN 可见全部租户数据，SALES 仅可见自己名下租户。**

**租户范围注入（核心）：** 分析请求进入服务层，按当前用户角色加载租户集合：
- `SALES`：从 `tenant_assignments` 查询其名下 `tenant_key` 集合；
- `ADMIN`：跳过租户过滤，可见全集。

**缓存策略：租户集合每次请求重新从 ADB 加载**（仅查 1 张表、索引命中，毫秒级），不做长缓存。这样 admin 修改分配后，商务**刷新页面或重新登录**即生效，无需主动失效缓存。

执行 SQL 前用 `NamedParameterJdbcTemplate` 绑定 `:tenantIds` 为该集合。前端传入的租户筛选与该集合**取交集**后再绑定——SALES 越权不可发生。

**租户注入强制校验（防越权兜底）：** SQL 执行器层强制校验所有分析查询必须包含租户过滤，防止遗漏：
1. 每个 `.sql` 文件头部必须声明 `-- @requires_tenant_filter`，未声明的 SQL 文件分析查询服务**拒绝执行**；
2. 执行前用 `NamedParameterJdbcTemplate` 绑定 `:tenantIds` 参数，若参数为空且角色非 ADMIN，**直接抛异常拒绝查询**；
3. 单元测试覆盖：每条分析 SQL 必须有「无 tenantIds 时拒绝」的测试用例。

**操作日志（operation_logs）：** 所有敏感操作写入日志表，便于审计追溯：
- `LOGIN` / `LOGOUT`：登录登出（含失败登录）。
- `CHANGE_PASSWORD`：密码修改。
- `CREATE_USER` / `DISABLE_USER`：创建/停用商务账号。
- `ASSIGN_TENANT` / `UNASSIGN_TENANT`：租户分配/移除（记录 tenant_key + 目标 user_id）。
- 日志只追加不删除，保留至少 90 天。

**SQL 注入防护：** 全部命名参数绑定，严禁拼接；`*.sql` 由受信任仓库管理；ADB 分析库账号只读。

### 7.4 前端工程规范与风格一致性保障

> 目标：每次发版后界面风格完全一致，杜绝"一个人一个写法"导致的风格漂移。

**1. Design Token（设计令牌）**

在 `tailwind.config.ts` 中集中定义所有主题变量，**禁止在组件中使用硬编码颜色值**：

```ts
// tailwind.config.ts
export default {
  theme: {
    extend: {
      colors: {
        // 主色系（与原型深色主题对齐）
        bg: { base: '#0F172A', card: '#1E293B', hover: '#334155' },
        text: { primary: '#F8FAFC', secondary: '#94A3B8', muted: '#64748B' },
        accent: { blue: '#3B82F6', green: '#10B981', purple: '#8B5CF6', amber: '#F59E0B' },
        border: '#475569',
      },
      borderRadius: { card: '12px', btn: '8px' },
      fontSize: { kpi: '28px', title: '16px', body: '13px', caption: '11px' },
    }
  }
}
```

所有组件引用 `bg-bg-card`、`text-text-primary` 等语义 class，**严禁内联 `style={{ color: '#xxx' }}`**。ESLint 加 `eslint-plugin-tailwindcss` 的 `no-custom-classname` 规则拦截非 Tailwind 类名。

**2. 组件库约束**

- 所有 UI 走 shadcn/ui 组件（Button / Dialog / Input / Table / DropdownMenu 等），不自造基础组件。
- ECharts 配置抽取为共享 `chartTheme.ts`（颜色数组、tooltip 样式、动画统一），所有图表引用同一份主题。
- 图表组件封装为独立 React 组件（`KpiCard.tsx` / `DiscountTable.tsx` / `TokenDistChart.tsx` / `FundTrendChart.tsx`），每个组件自包含样式与配置。

**3. 构建产物缓存策略**

Vite 构建文件名带 content hash（`assets/[name].[hash].js`），Nginx 配置：
- `assets/*` 文件设 `Cache-Control: max-age=31536000, immutable`（一年强缓存）；
- `index.html` 设 `Cache-Control: no-cache`（每次请求校验）。

确保发版后用户立即加载新资源，不会出现新旧混合。

**4. 单图表容错**

每个图表独立请求后端 API，单个图表查询失败时：
- 该卡片显示错误态 + 重试按钮，**不影响其他图表**；
- 使用 TanStack Query 的 `isLoading` / `isError` / `data` 三态渲染骨架屏 / 错误态 / 数据。

**5. 环境管理**

| 环境 | 前端 | 后端 | ADB |
|------|------|------|-----|
| dev（开发） | 本地 Vite dev server | 本地 Spring Boot | dev 账号（`app_rw` + `analytics_ro`） |
| prod（生产） | Nginx 静态托管 | java -jar / Docker | prod 账号（`app_rw` 读写 3 表 + `analytics_ro` 只读分析表） |

通过环境变量切换，**不硬编码连接信息**：

```
# 后端
SPRING_PROFILES_ACTIVE=prod
ADB_URL=jdbc:mysql://adb-host:3306/maas
ADB_APP_USER=app_rw
ADB_APP_PASS=***
ADB_ANALYTICS_USER=analytics_ro
ADB_ANALYTICS_PASS=***
ADB_QUERY_TIMEOUT=10          # 分析查询超时（秒），JdbcTemplate.setQueryTimeout

# 前端
VITE_API_BASE=/api            # 同源部署下默认 /api
```

**6. ADB 查询超时**

分析查询通过 `JdbcTemplate.setQueryTimeout(10)` 设置 10 秒超时，防止慢查询拖垮整个页面。超时后该图表显示错误态 + 重试按钮（与单图表容错机制配合）。

### 7.5 日志与可观测

> 目标：生产出问题时能快速定位，不为"不知道谁查了什么"丢人。

- **后端请求日志**：每个 API 请求记录 `时间 · 用户ID · 角色 · 接口路径 · 耗时 · 状态码`，INFO 级别落文件。
- **分析查询日志**：每次执行命名 SQL 记录 `queryName · 注入的 tenantIds 数量 · 耗时 · 返回行数`，便于排查慢查询。
- **慢查询告警**：ADB 查询超过 5s 记 WARN 日志，超过 15s 记 ERROR。
- **前端错误上报**：未捕获的 JS 异常通过 `window.onerror` 上报到后端 `/api/log/error`（简单落库或落文件，不引入 Sentry 等外部服务）。
- **不引入**：分布式链路追踪、ELK、Prometheus——当前规模不需要。

## 8. API 设计

- **认证**：`POST /api/auth/{login,refresh,logout}`、`GET /api/auth/me`、`POST /api/auth/change-password`（校验旧密码 → 更新新密码 → 置 `must_reset=FALSE` → 清 cookie 强制重登）
- **管理端（ADMIN）**：
  - `GET /api/admin/users`（用户列表，含 last_login_at / enabled 状态）
  - `POST /api/admin/users`（新建商务）：
    - 请求体：`{ display_name: "张三", region: "华东区", role: "SALES", password: "初始密码" }`
    - 后端自动生成 `username`（姓名全拼 + 重名自增），bcrypt 加密密码，置 `must_reset=TRUE`
    - 返回体：`{ id, username, display_name, region, role, must_reset }`（**返回生成的 username 供 admin 确认**）
    - 写入 `operation_logs`（action=CREATE_USER）
  - `PUT /api/admin/users/{id}`（更新，如修改 region / 重置密码）
  - `DELETE /api/admin/users/{id}`（停用，`enabled=false`，不删数据）
  - `GET /api/admin/users/{id}/tenants`（查已分配租户列表）
  - `PUT /api/admin/users/{id}/tenants`（整体替换分配集合）：
    - 请求体：`{ tenants: ["te-xxx", "te-yyy"] }`（tenant_key 数组）
    - 后端事务内执行：先删除该用户旧分配 → 再批量插入新分配
    - **互斥校验**：插入前检查每个 tenant_key 是否已被其他用户占用（`UNIQUE(tenant_key)` 约束兜底），冲突时返回 409 + 冲突租户列表
    - 每次分配/移除写入 `operation_logs`（action=ASSIGN_TENANT / UNASSIGN_TENANT）
  - `GET /api/admin/tenants`（从 ADB 实时查可分配租户列表，**排除已被分配的租户**）
    - 后端先查 ADB 全量 `SELECT DISTINCT tenant_id, tenant_name`，再查 `tenant_assignments` 已占用的 tenant_key 集合，取差集返回
- **分析端（USER）**：
  - `GET /api/analytics/tenants`（我可访问的租户，供筛选下拉）
  - `POST /api/analytics/query/{queryName}`（按命名 SQL + 筛选取数，服务端注入租户范围）
    - 请求体：`{ startDate, endDate, tenants?, page?, pageSize?, sortBy?, sortOrder? }`（tenants 可选，服务端与名下租户取交集）
    - **queryName 白名单**：只允许从 `resources/sql/*.sql` 加载，禁止路径穿越（`../` 等），非法 queryName 返回 404
    - **表格排序**：card 121/138 支持 `sortBy`（列名）+ `sortOrder`（asc/desc），默认 card 121 按消费金额降序、card 138 按租户名升序
    - 表格类查询支持 `page` + `pageSize` 分页，默认返回前 100 行；KPI/图表类查询不分页
- **系统**：
  - `GET /api/health`（健康检查，返回 `{ status: "UP", db: "UP" }`，用于容器探活）

> 无 dashboards 配置接口——布局前端硬编码，查询 SQL 在代码仓库，走 Git + 部署。

## 12. 风险与对策

| 风险 | 对策 |
|---|---|
| 无状态 JWT 不能即时吊销 | access 短 TTL（30min）+ 账号禁用阻止新签发；密码修改后清 cookie 强制重登 |
| 租户范围注入遗漏越权 | `-- @requires_tenant_filter` 标记强制校验 + 执行器拒绝无 tenantIds 查询 + 单测覆盖；ADB 分析库只读兜底 |
| 租户分配竞态（并发分配同一租户） | `tenant_assignments.tenant_key` 全局 UNIQUE 约束兜底 + API 层先查后写 + 冲突返回 409 |
| admin 改分配后商务缓存不生效 | 租户集合每次请求重新从 ADB 加载，刷新即生效（§6） |
| ADB 大查询慢 | 预聚合宽表 + 查询超时 10s（§7.4.6）+ 前端单图表容错 |
| 前端发版后浏览器缓存旧资源 | Vite hash 文件名 + Nginx immutable 长缓存 + index.html no-cache（§7.4） |
| ADB 无租户维度表（tenant_name 缺失） | P0 先核实 ADB 是否有 tenant 维度；否则分配 UI 用 tenant_key + 手填别名 |
| 时区不一致导致数据口径偏差 | 前端按北京时间传参，SQL 内部 `CONVERT_TZ` 转换（§5.3 时区处理） |
| 风格漂移（多次发版后不一致） | Design Token 集中管理 + shadcn/ui 组件 + 共享 chartTheme + ESLint 拦截硬编码色值（§7.4） |
| ADB 作为 OLTP 写入性能（3 张应用表） | ADB 是 OLAP 引擎，单行写入延迟可能高于 OLTP 库；本系统写入量极低（admin 分配/账号管理/日志），可接受；P0 验证实际延迟，若不达标再引入独立 MySQL |
