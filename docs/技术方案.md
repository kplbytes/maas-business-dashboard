# MaaS 商务看板系统 · 技术方案

> 版本：v0.2（决策已确认）　日期：2026-07-16
> 定位：**轻量级看板 + 权限控制**。只做两件事——按账号隔离租户数据、展示经营看板。

## 1. 背景与目标

MaaS 平台现有 Metabase 看板（dashboard/14）无法按登录账号做租户级行权限隔离。本系统自建一套轻量看板，让商务同学登录后只看到**自己名下租户**的数据，默认看名下全部租户汇总，与 Metabase 口径对齐。

**目标：**

1. 商务登录默认看「我名下所有租户的汇总」，支持租户/时间筛选。
2. 管理员可对每个商务的租户列表做增删改查。
3. 数据隔离强一致：商务只能看到被分配的租户，越权不可发生。
4. 支持多种图表，整体风格对标 Metabase。
5. **极简**：DB 只存「登录 + 商务-租户关系」，看板/查询/配置全在代码里，变更靠改代码+部署。

## 2. 解决思路

权限层从 Metabase 搬到**自建后端**——后端是唯一信任边界。所有分析查询执行 SQL 前，服务端强制把「当前用户名下租户集合」注入为 `WHERE tenant_id IN (:myTenants)`。前端传入的租户筛选只与该集合取交集，绝不作为唯一过滤源。Metabase 仅作设计参考，运行时不依赖。

## 3. 总体架构

```
┌───────────────────────────────────────────────────────────────┐
│  浏览器  React SPA (Vite + TS + shadcn/ui + Tailwind + ECharts) │
│  登录 / 看板 / 后台管理 / 筛选                                   │
└──────────────────────────┬────────────────────────────────────┘
                           │ HTTPS · JWT (httpOnly cookie)
┌──────────────────────────▼────────────────────────────────────┐
│  Spring Boot 后端 (Java 17)                                    │
│  ┌──────────┐ ┌──────────┐ ┌─────────────┐ ┌───────────────┐  │
│  │ Auth/JWT │ │ RBAC 守卫 │ │  管理服务    │ │ 分析查询服务   │  │
│  │ (无状态)  │ │          │ │ 用户/分配CRUD│ │ 命名SQL+租户注入│ │
│  └──────────┘ └────┬─────┘ └──────┬──────┘ └───────┬───────┘  │
│                    │              │                 │          │
│             ┌──────▼──────┐  看板配置=代码          │          │
│             │ 应用状态库    │  (dashboard.yml+*.sql) │          │
│             │ 仅 2 张表     │                       │          │
│             │ SQLite / PG  │              ┌────────▼────────┐ │
│             └─────────────┘               │ ADB-MySQL (只读) │ │
│                                           │ tenant_id IN     │ │
│                                           │ (:myTenants)     │ │
│                                           └─────────────────┘ │
└───────────────────────────────────────────────────────────────┘
```

**两条通路：** 应用状态（2 表，OLTP）与分析数据（ADB 只读）。看板/查询配置不在 DB，在代码仓库。

## 4. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 前端 | React 18 + Vite + TS + **shadcn/ui + Tailwind** + ECharts | shadcn 主题完全可控，admin 与看板共用 Metabase 令牌，风格一致 |
| 前端状态 | TanStack Query（服务端态）+ Zustand（客户端态） | 不重复缓存 |
| 后端 | Java 17 + Spring Boot 3.x | Spring Security 认证/RBAC、Spring Data JPA（2 表）、JdbcTemplate/MyBatis（ADB） |
| 认证 | Spring Security + JWT (jjwt) + bcrypt，**无状态** | 见 §6 |
| 应用状态库 | **SQLite（起步）/ PostgreSQL（生产多实例）** | 仅 2 张表，SQLite 零基建 |
| 分析库 | 阿里云 AnalyticDB for MySQL（只读账号） | mysql-connector-j，MySQL 方言 |
| 迁移 | Flyway | 建 2 表 + 后续 ALTER |
| 看板/查询配置 | `dashboard.yml` + `resources/sql/*.sql`（代码仓库） | 版本可控，改图改部署 |

## 5. 数据模型（极简：2 张表）

### 5.1 应用状态库

```
users
  id            BIGINT PK
  username      VARCHAR(64)  UNIQUE      -- 登录名
  password_hash VARCHAR(255)             -- bcrypt
  display_name  VARCHAR(128)
  role          VARCHAR(16)              -- ADMIN / SALES（枚举列，仅两角色）
  enabled       BOOLEAN
  created_at    TIMESTAMPTZ
  updated_at    TIMESTAMPTZ

tenant_assignments
  id            BIGINT PK
  user_id       BIGINT  FK→users.id
  tenant_key    VARCHAR(64)              -- 注入 ADB 的 tenant_id 值
  tenant_name   VARCHAR(128)             -- 冗余存展示名（分配时从 ADB 带回）
  assigned_by   BIGINT  FK→users.id
  assigned_at   TIMESTAMPTZ
  UNIQUE(user_id, tenant_key)
```

> 角色用 `users.role` 枚举列，不再单独建 roles/user_roles 表。租户主数据**不建表**——分配时 admin 从 ADB 实时查 `SELECT DISTINCT tenant_id, tenant_name` 选取，写入 `tenant_assignments` 时把 key+name 一起存。

### 5.2 分析层（ADB-MySQL，只读）

直连 ADB，执行 `resources/sql/*.sql` 命名查询。建议在 ADB 内预聚合宽表（按 `tenant_id/model/date/hour`）保证秒级响应；今日数据走实时表 + 「数据可能滞后」水印。

### 5.3 看板/查询配置（代码，非 DB）

```
resources/
  sql/
    token_trend.sql          -- 含 :tenantIds :startDate :endDate 占位
    revenue_trend.sql
    api_calls.sql
    model_mix.sql
    top_tenants.sql
    ...
  dashboards.yml             -- 看板布局与 widget 定义
```

`dashboards.yml` 示例：
```yaml
dashboards:
  - id: customer-ops
    name: 客户经营看板
    widgets:
      - id: token-trend
        title: Token 消耗趋势
        query: token_trend
        chart: line
        params: [date_range, granularity]   # tenants 由服务端强制注入
        layout: { x: 0, y: 0, w: 8, h: 1 }
      - id: model-mix
        title: 模型用量分布
        query: model_mix
        chart: donut
        ...
```

新增/改图表 = 加一个 `.sql` + 改 `dashboards.yml` + 重新部署。前端 `GET /api/dashboards/{id}` 拿这份配置渲染。

## 6. 权限与安全模型

**认证（无状态 JWT，无 token 表）：**
- 登录签发 access (30min) + refresh (7d)，均为无状态 JWT，存 **httpOnly SameSite=Strict cookie**。
- 刷新：refresh JWT 验签通过即签新 access；不落库。
- 登出：清 cookie。
- **撤销**：无状态 JWT 不能即时吊销；靠①access 短 TTL（30min 内自然失效）②`users.enabled=false` 阻止新 token 签发。内部工具可接受；若未来需即时踢人再加黑名单表。

**RBAC：** `role=ADMIN` / `role=SALES`，方法级 `@PreAuthorize`。管理端要求 ADMIN，分析端要求已认证。

**租户范围注入（核心）：** 分析请求进入服务层，按当前用户从 `tenant_assignments` 加载 `tenant_key` 集合（缓存于 SecurityContext）；执行 SQL 前用 `NamedParameterJdbcTemplate` 绑定 `:tenantIds` 为该集合。前端传入的租户筛选与该集合**取交集**后再绑定——SALES 越权不可发生，ADMIN 可见全集。

**SQL 注入防护：** 全部命名参数绑定，严禁拼接；`*.sql` 由受信任仓库管理；ADB 账号只读。

## 8. API 设计

- **认证**：`POST /api/auth/{login,refresh,logout}`、`GET /api/auth/me`
- **管理端（ADMIN）**：
  - `/api/admin/users`（GET 列表 / POST 新建 / PUT 更新 / DELETE 停用）
  - `/api/admin/users/{id}/tenants`（GET 查已分配 / PUT 整体替换分配集合）
  - `/api/admin/tenants`（GET，从 ADB 实时查可分配租户列表）
- **分析端（USER）**：
  - `GET /api/analytics/tenants`（我可访问的租户，供筛选下拉）
  - `GET /api/dashboards[/{id}]`（读代码配置的看板定义）
  - `POST /api/analytics/query/{queryName}`（按命名 SQL + 筛选取数，服务端注入租户范围）

> 看板/查询配置无 CRUD 接口——配置在代码仓库，走 Git + 部署。

## 9. 已确认决策

| 决策 | 选择 | 说明 |
|---|---|---|
| A 应用状态库 | **DB 仅 2 表（users + tenant_assignments），其余进代码** | 轻量；租户主数据从 ADB 实时取 |
| B 前端 UI | **shadcn/ui + Tailwind** | 风格一致、主题可控 |
| C 配置维护 | **代码 + SQL 文件 + Flyway 迁移** | 版本可控，变更改代码部署 |

## 11. 扩展性

- **新图表/新看板**：加 `.sql` + 改 `dashboards.yml` + 部署，不发版不改 DB。
- **新筛选维度**：SQL 加占位 + widget `params` 声明。
- **新角色**：`users.role` 扩枚举 + `@PreAuthorize`。
- **数据源扩展**：`QueryExecutor` 接口抽象，未来加第二数据源不影响上层。

## 12. 风险与对策

| 风险 | 对策 |
|---|---|
| 无状态 JWT 不能即时吊销 | access 短 TTL + 账号禁用阻止新签发；必要时加黑名单表 |
| 改图表需重新部署 | 已确认接受（轻量优先）；部署流程自动化降低成本 |
| ADB 大查询慢 | 预聚合宽表 + 结果缓存 + 查询超时 |
| 租户范围注入遗漏越权 | `requires_tenant_filter` 在 SQL 执行器强制校验，单测覆盖；ADB 账号只读兜底 |
| ADB 无租户维度表（tenant_name 缺失） | P0 先核实 ADB 是否有 tenant 维度；否则分配 UI 用 tenant_key + 手填别名 |
| 今日数据时区/计费延迟 | 涉今日图表加「数据可能滞后」水印 |
